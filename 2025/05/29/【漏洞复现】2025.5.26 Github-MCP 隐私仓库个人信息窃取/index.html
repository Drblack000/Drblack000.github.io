【漏洞复现】2025.5.26 Github-MCP 隐私仓库个人信息窃取

发表于2025-05-29|更新于2025-06-21
|字数总计:2.8k|阅读时长:14分钟|阅读量:

漏洞描述

GitHub MCP 集成（在 GitHub 上有 1.4 万颗星）的关键漏洞。该漏洞允许攻击者通过恶意 GitHub Issue 劫持用户的代理，并强制其泄露私有仓库数据。

官方攻击流程图

漏洞复现

分别创建一个私有仓库 personal_info和公有仓库 mcp_test_public

创建完成后，在私有仓库personal_info添加一个模拟的敏感文件readme.md用于测试

配置github

首先需要在设置-开发者设置-个人访问令牌处申请令牌

配置完成后，进行权限修改，将仓库的读写权限打开，复制对应的key。

配置cursor

添加下面的配置文件，填上自己的key

先选用cursor作为客户端进行测试

Github-mcp功能检测

先检测功能是否可用。

读取权限检测

读取一下私有仓库的文件内容，发现成功读取

写入权限检测

其他客户端Cherrystudio也类似配置即可

提交issue

提交下面的恶意内容

攻击测试

漏洞原文prompt：Take a look at the issues in my open source repository xxx and fix them!

实际测试出来有一个问题

他缺少上下文，可能无法直接用github_mcp工具。改进一下

prompt：使用github_mcp先查询我的github名称，确认我是谁后，Take a look at the issues in my open source repository mcp_test_public and use github_mcp fix them!

Cursor -半自动复现

复现效果和模型关系很大，cursor由于没有会员不能使用原始漏洞文章提到的的Claude模型，cursor复现只能说是半成功吧。

由于涉及到的隐私内容，他会询问一下是否愿意继续操作。回复了yes,i do后，成功完成漏洞复现(半复现)

内容是base64加密的（因为中间github_mcp调用update_file默认缺少了一个sha参数，失败了很多次，他就尝试用加密的方式传输，实际上提醒他加上sha参数即可），解密一下即可

这里提一下，使用cursor有一个好处：如下图，cursor默认设置了每次调用mcp进行问询，可以查看mcp请求内容（在一定程度上防止这类隐蔽的攻击，但很多人会配置自动调用mcp不问询，毕竟mcp调用多次的时候，挨个去点击还是比较麻烦）

cherrystudio-完全复现

采用cherrystudio+qwen-max复现成功

前半部分prompt用于确保模型可以正确定位到工具使用，以及账户信息，避免无效沟通。

## 防御措施
该漏洞和**<font style="color:rgb(25, 27, 31);"> GitHub MCP 服务器代码本身没有关系，而是架构层面引起的问题。</font>**

**<font style="color:rgb(25, 27, 31);">1. 采用精确权限控制</font>**

<font style="color:rgb(25, 27, 31);">使用这类MCP时，遵循最小权限原则。避免过大的权限导致任意的仓库可读可写。</font>**<font style="color:rgb(25, 27, 31);">这不是 GitHub MCP 服务器代码本身的缺陷，</font>**<font style="color:rgb(25, 27, 31);">所以 GitHub 无法通过服务器端补丁解决此漏洞。</font>

**<font style="color:rgb(25, 27, 31);">2. 使用安全监控</font>**

<font style="color:rgb(25, 27, 31);">官方建议：部署专业的安全扫描器，例如 Invariant 的 MCP-scan，以持续审计代理和 MCP 系统之间的交互。</font>
